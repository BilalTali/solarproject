# CHAPTER 1: THE FOUNDATIONAL 5-TIER HIERARCHY

The PM Surya Ghar application relies on a strictly enforced, vertically integrated 5-tier architecture. It segregates duties across financial management, clerical validation, regional leadership, active sales, and ground-level data collection. This structure prevents fraudulent data entry, ensures proper financial disbursement, and gamifies the sales process while protecting the immutable ledger.

## 1.1 The Chain of Command
1. **System Administrator (Admin)** - Unrestricted global control.
2. **Operator (Backend Support Staff)** - Sibling to the Admin with restricted read/write access focused purely on application processing.
3. **Business Development Manager (BDM / Super Agent)** - The apex field role [SSM-YYYY-XXXX]. Validates data, earns overrides, and absorbs field points.
4. **Business Development Executive (BDE / Agent)** - The primary field worker [SM-YYYY-XXXX]. Advances statuses, redeems points, and manages field teams.
5. **Field Enumerator (ENM)** - The isolated baseline data collector [ENM-YYYY-XXXX]. Solely focused on lead submission with absolute financial compartmentalization.

# CHAPTER 2: SYSTEM ADMINISTRATOR (ADMIN)
**Position in Hierarchy:** Pinnacle Authority
**Authentication Security:** Email-OTP mechanism (10-minute expiry window, 5-minute lockout upon failure). 

The System Administrator role serves as the singular authority over the structural, financial, and regulatory boundaries of the platform. The Admin does not typically perform day-to-day data entry but rather oversees the automated and clerical operations below them.

## 2.1 Absolute Lead Pipeline Mastery
While field agents are locked into strict forward-only progression rules, the Admin account possesses omnipotent database write privileges across the entire `leads` table. 
- **Terminal Status Resolution:** Admins are the only users (along with Operators) who can finalize a lead. They transition leads from `installed` up to the final MNRE-level phases: `subsidy_applied` and ultimately `completed`. The `completed` status is critical as it serves as the mechanical trigger that generates commission records in the database.
- **Exception Handling & Rejection:** Admins have the sole authority to mark a lead as `invalid` (fraudulent or garbage data), `duplicate` (already exists in the discom limits), `on_hold` (pending external validation), or `rejected`. 
- **Bulk Orchestration (Orphaned Leads):** When citizens apply via the public form without a valid referral code, or if a field agent is soft-deleted, their leads fall into the `admin_pool`. The Admin utilizes the `/api/v1/admin/leads/bulk-action` endpoint to mass-reassign these orphaned leads to active performing agents, instantly transferring ownership and pipeline tracking.
- **Verification Overrides:** The normal verification chain requires a BDM to approve a BDE's lead. However, if a lead enters an `admin_override` state (which automatically occurs if a BDM reverts a lead 3 times to prevent infinite loops), the Admin steps in to manually force verification.

## 2.2 Financial Controller (Commissions & Withdrawals)
The Admin is the central bank of the Surya Ghar platform. The platform operates on a "Double-Entry" ledger that tracks both what is owed and what is paid.
- **Commission Slab Configuration:** Admins control the `commission_slabs` table. They actively map out the capacity ranges (1kW to 10kW+) to the specific Indian Rupee (₹) payout split required for the BDE and the BDM override. For example, setting 10kW to yield ₹5,500 for the Agent and ₹1,200 for the Super Agent.
- **Wallet Credit Authorization:** When a lead hits `completed`, a commission record is generated in an `unpaid` or `pending_approval` state. The Admin must review this ledger. By pressing 'Approve', the Admin triggers a transactional database lock (`DB::transaction` with `lockForUpdate()`) that forcefully credits the exact ₹ amount into the respective agent's `wallet_balance`.
- **Withdrawal Processing Lifecycle:** Agents request payouts to their bank accounts. These land in the Admin portal as `pending` withdrawals. The Admin reviews the AES-256 encrypted banking details (Aadhaar, IFSC, Account Number). The Admin then physically executes the bank transfer outside the system. Returning to the portal, the Admin marks the request as `processed` (which permanently debits the agent's digital wallet) or `rejected` (which requires submitting a mandatory `rejection_reason` string back to the agent).

## 2.3 The Gamification Engine (Offers & Points Architecture)
To drive sales, the system utilizes a high-engagement gamification engine that only the Admin can configure and monitor.
- **Offer Construction:** Admins create limited-time "Offers" (e.g., "The Diwali Solar Dash"). They define the `target_points` required, the exact temporal boundaries (`offer_from` to `offer_to`), and the `grace_period_days`. 
- **Points Equivalence Settings:** Admins map physical solar capacities to abstract gamification points via the dynamic `capacity_points_json` variable (e.g., dictating that a 3kW installation equals 1.0 point, while a 10kW installation equals 5.0 points).
- **Collective Offers:** Admins can launch "Collective" targeted offers where progression is shared. In these scenarios, the Admin manually dictates prize disbursements based on their offline analysis of leaderboard participants, rather than automatic point thresholds.

## 2.4 Security, Credentialing, and Auditing
- **Total User Authority:** The Admin can approve new incoming registrations, temporarily `suspend` accounts under investigation for fraud, or permanently `reject` / soft-delete accounts.
- **Credential Revocation (The Kill Switch):** If an agent goes rogue, the Admin can trigger the "Revoke" feature on the agent's Joining Letter. Mechanically, this sets `letter_revoked = true` in the database, nullifies their valid authorization, and importantly, instantly destroys all active Sanctum tokens, forcefully ejecting the user from all active mobile and desktop sessions.
- **QR Code Anomaly Logging:** Every PDF iCard features a SHA-256 encrypted QR Code. When scanned by citizens, the request logs the IP address and User Agent to the `qr_scan_logs` table. Admins monitor this list to ensure iCards aren't being photocopied and scanned in impossible geographic locations.

## 2.5 Branding, Marketing, and Public Engagement
- **Dynamic Configuration:** Admins modify the `settings` table directly affecting the public UI. This includes `company_full_name`, support contact lines, overlay texts on the login screen, and strict numerical rules like `doc_expiry_electricity_bill` (e.g., 90 days).
- **Testimonial Curation:** When a solar installation finishes, a signed URL is sent to the beneficiary to collect a survey. These surveys (1 to 5-star ratings with photos) land in the Admin dashboard. The Admin curates these and toggles the `is_published` boolean, seamlessly broadcasting the best testimonials to the `/public` routes.

# CHAPTER 3: OPERATOR (BACKEND SUPPORT STAFF)
**Position in Hierarchy:** Lateral to Admin (Clerical Wing)
**Role Enum:** `operator`

As the platform scales across hundreds of agents and thousands of citizens, the System Administrator cannot manually review every Aadhaar card and electricity bill. Thus, the Operator role exists as a dedicated clerical, non-financial subset of the Admin console.

## 3.1 Unrestricted Global Pipeline Visibility
- **Lead Access:** Unlike field agents who are strictly blind to leads outside their specific downline, the Operator shares the Admin's panoptic view. They can query, filter, search, and parse every single lead across the `admin_pool`, `super_agent_pool`, and `agent_pool`.
- **Status Advancement Operations:** The Operator handles the heavy, daily bureaucratic lifting. When a lead needs to be moved from `installed` to `subsidy_applied` after matching government portal records, the Operator executes this. They process rejections and mark duplicates just like the Admin. 

## 3.2 Strict Matrix Constraints and Exclusions
The genius of the Operator role is what they are restricted from seeing. The system is designed under the principle of Least Privilege. 
- **The Financial Firewall:** Operators have absolute ZERO access to the `commissions`, `wallet_transactions`, and `withdrawal_requests` architecture. They cannot see how much an agent earns, they cannot approve payouts, and they cannot manipulate commission slabs. If they attempt to hit `/api/v1/admin/commissions`, the server returns a 403 Forbidden.
- **The Settings Firewall:** Operators cannot modify the `settings` table. They cannot change points, branding, or gamification mechanics.
- **The User Firewall:** Operators cannot suspend, soft-delete, or approve Business Development Managers or Executives. They do not hold the keys to user termination.
- **The Leaderboard Exclusion:** Operators do not participate in or view the competitive leaderboards, shielding them from gamification biases.

# CHAPTER 4: BUSINESS DEVELOPMENT MANAGER (BDM / SUPER AGENT)
**Position in Hierarchy:** Tier 1 Field Leadership
**Identifier/Code format:** `SSM-YYYY-XXXX`

The Business Development Manager is the regional apex predator of the field force. They are not expected to walk door-to-door (though they can). Instead, their primary architectural mandate is network expansion, team supervision, and first-line document validation. 

## 4.1 Team Scope and Hierarchical Sovereignty
- **Downline Visibility:** A BDM has a customized scope that filters the massive global database down to only their direct operational tree. They can see every Business Development Executive (BDE) assigned to their `super_agent_id`, and by extension, the Field Enumerators (ENMs) spawned by those BDEs. 
- **Direct Lead Submission Guarantee:** BDMs can submit their own leads directly into the system. Because they represent regional authority, these direct submissions bypass standard verification layers, landing in the `super_agent_pool` fully authorized and `auto-verified`. The BDM can then manually assign these "house leads" to a struggling downline BDE to help them meet targets, transitioning the lead to the `agent_pool`.

## 4.2 The Verification Mandate (The First Line of Defense)
The BDM acts as the critical quality-control filter before data reaches the Admin/Operator.
- **Pending Verification Queue:** When a downline BDE submits an application, the lead is locked into the `pending_super_agent_verification` status. 
- **The Review Process:** The BDM must manually inspect the uploaded images (Electricity bills, Aadhaar cards). 
- **The Revert Engine:** If the BDM detects a blurry photo or incorrect consumer number, they issue a "Revert" command. The system logs the rejection reason and pings the BDE via FCM Push Notification (`lead_reverted`). 
- **Auto-Escalation Safeguard:** To prevent infinite loops of incompetence where a BDE continually uploads bad documents and the BDM continually reverts, the system tracks `revert_count`. On the 3rd revert, the lead is yanked from the BDM's jurisdiction and escalated to `admin_override`.

## 4.3 The Financial Pipeline (Override Commissions)
The BDM is rewarded for building and managing a highly productive network through a robust financial architecture.
- **Override Structure:** When a downline BDE successfully closes a lead (lead hits `completed`), the system generates two separate commission records (Double-Entry workflow). One for the BDE, and an "override" commission for the BDM (e.g., BDE earns ₹3000, BDM earns an override of ₹700).
- **Data Entry Duty:** BDMs are actively responsible for entering commission requests into the system for their downline team. This generates an `agent_entered` or `both_entered` flag, queuing the paperwork for Admin approval.
- **The 24-Hour Immutable Lock:** To prevent historical accounting fraud, whenever a BDM enters a commission, a scheduled server command (`commissions:lock-expired`) monitors it. Exactly 24 hours after creation, the row is irrevocably stamped with a `locked_at` timestamp. Once locked, the BDM can never edit or alter the financial details again.

## 4.4 Gamification "Absorption" Mechanics
One of the most powerful system mechanics reserved exclusively for the BDM involves the gamified Offer system.
- **The Scenario:** The Admin launches an offer requiring 100 points. A downline BDE grinds hard but only reaches 92 points when the deadline `offer_to` strikes. 
- **Grace Period Monitoring:** The system enters a `grace_period_days` holding pattern.
- **The Absorption Event:** Once the grace period officially terminates, the daily CRON job (`offers:process-expired`) sweeps the database. Because the BDE fell short and failed to redeem, the 92 points do not simply vanish. Instead, the points are "Absorbed" directly upward into the BDM's profile as a massive performance bonus (`points_absorbed` notification), enriching the BDM for driving their team close to the target.

# CHAPTER 5: BUSINESS DEVELOPMENT EXECUTIVE (BDE / AGENT)
**Position in Hierarchy:** Tier 2 Direct Field Sales
**Identifier/Code format:** `SM-YYYY-XXXX`

The BDE is the primary kinetic force of the PM Surya Ghar application. They are the door-to-door closers, the data entry specialists, and the active participants in gamified sales challenges. Their permissions prioritize forward momentum and strict data compliance.

## 5.1 The Frictionless Data Collection Mechanics
- **Lead Generation:** BDEs generate leads mapped to `agent_submission`. They upload critical KYC assets (Aadhaar, Bank Passbook, Roof Photos) which are subjected to immediate AES-256 field encryption upon storage.
- **Telemetry and Call Logging:** The portal expects high-volume interactions. BDEs must aggressively log interactions using the `lead_call_logs` table. They are restricted to a highly specific taxonomy of outcomes: `no_answer`, `callback_requested`, `interested`, `not_interested`, `wrong_number`, `already_registered`, and `disconnected`. 
- **The "Stale Lead" Whip:** If a BDE ignores a lead, the system watches. A daily scheduled command (`leads:remind-stale`) fires at 09:00 AM. If a lead has lacked a call log for 72 hours, a severe push notification strikes the devices of both the BDE and their supervising BDM, forcing accountability.

## 5.2 The Immutable Forward-Only Pipeline
To maintain data integrity and prevent manipulation of conversion statistics, BDEs operate under a rigid state machine regarding lead statuses.
- **Forward Only Policy:** A BDE can push a lead's status forward: `contacted` → `site_survey` → `registered` → `installed`. 
- **Hard Exclusions:** A BDE physically cannot push a lead backward. If an error is made, they must contact Support/Operators. Furthermore, BDEs cannot execute terminal states. They cannot mark a lead as `completed`, nor can they mark it `rejected`, `invalid`, or `on_hold`. Only higher authorities can terminate an application's lifecycle.

## 5.3 The Re-Upload State Machine Protocol
What happens when a BDE uploads a blurry Aadhaar card and the BDM hits 'Revert'?
1. The BDE receives a push notification identifying the failure.
2. The BDE navigates to the specific document endpoint `/api/v1/agent/leads/{id}/documents/reupload`.
3. Upon uploading the new image file, the old row in `lead_documents` is not deleted (to preserve audit history). Instead, it is flagged with the crucial `is_superseded = true` boolean.
4. The verification status of the lead is forcefully reset to `pending_super_agent_verification`.
5. The pipeline status remains right where it was, preventing the agent from losing their place in the forward-momentum sequence.

## 5.4 Downline Construction (Field Enumerator Generation)
A highly successful BDE cannot knock on every door in Jammu & Kashmir themselves. Therefore, BDEs are granted the power to create Sub-Users exclusively in the form of Field Enumerators (ENMs).
- BDEs approve new ENM accounts.
- BDEs monitor the exact pipeline generation of their ENMs.
- BDEs maintain full visibility of their localized team structure but cannot spawn other BDEs or BDMs.

## 5.5 Personal Gamification and Redemption Tactics
The BDE is the primary target of the Admin's Offer gamification system.
- **Live Tracking:** Subscribing to Tanstack React real-time queries, BDEs watch their `total_points` increment as leads move to `completed`.
- **The Redemption Strike:** BDEs monitor the delta between `total_points` and `redeemed_points`. When the delta crosses the `target_points` threshold, their UI unlocks a 'Redeem' action. Triggering this creates an `offer_redemptions` row and sends an `offer_redemption_pending` ping directly to the Admin.
- **Leaderboard Metrics:** The BDE battles other agents on the monthly `/api/v1/agent/leaderboard/monthly`. Their physical rank is computed by tallying **completed leads in the current month**, using **total commission amount** heavily as a tie-breaker. 

## 5.6 Operational Credentials
The BDE is a fully authorized platform representative. They maintain the power to constantly generate deeply verified, cryptographically signed API URL endpoints that download highly formatted PDF versions of their **Official iCard** and their **Official Joining Letter**. These URLs securely expire within 30 minutes to prevent unauthorized broad distribution.

# CHAPTER 6: FIELD ENUMERATOR (ENM)
**Position in Hierarchy:** Tier 3 Ground Data Collection
**Identifier/Code format:** `ENM-YYYY-XXXX`

The Field Enumerator is the absolute bare-metal implementation of a user in the Surya Ghar system. The Enumerator role strips away gamification, strips away team management, and strips away status manipulation, leaving only two functions: Submitting raw data and getting paid directly.

## 6.1 The Bare-Metal Interface (What They Can Do)
- **Raw Lead Generation:** The ENM logs in. The UI is incredibly simplified. They submit leads bearing the `enumerator_submission` tag.
- **Immediate Escrow:** The moment an ENM hits submit, the lead instantly routes upwards into the parent BDE's administrative pool. The ENM loses all write-access to the record post-submission.
- **Read-Only Persistence:** The ENM is permitted to view a list of leads they have authored to track whether they are progressing through the system, but they are physically blocked from altering a status, uploading superseded documents, or logging call metrics.

## 6.2 The "Anti-Split" Absolute Financial Independence Rule
To ensure ground workers are not financially exploited by their supervising BDEs, the systemic architecture enforces an absolute routing exception for Enumerator commissions.
- **Standard Protocol:** Usually, a lead triggers a BDE commission and a BDM override.
- **The Enumerator Override (Rule R.10):** If the lead `source` is an ENM, the internal `CommissionService` engine rewrites the routing map. The commission is routed **EXCLUSIVELY** to the Field Enumerator. 
- **Zero-Sum for BDE:** The supervising BDE, despite handling the call logs and status advancements for that specific lead, earns exactly ₹0.00 for the installation. This enforces the guarantee that ENM flat fees are never cut or shared.
- **Wallet Autonomy:** Because they earn independently, ENMs utilize their own personal digital Wallet space. They pull up read-only views of their ledger and autonomously submit withdrawal requests directly to the Admin without needing permission from their parent BDE.

## 6.3 Strict Architectural Rejections (The Exclusions Matrix)
The ENM represents the frontier edge of application security. As entry-level participants, they face massive server-side 403 blocks to prevent misrepresentation.
- **The Credential Ban:** Field Enumerators are strictly, explicitly forbidden from generating PDF iCards. They are completely, explicitly forbidden from generating PDF Joining Letters. Any attempt to hit the backend `/api/v1/enumerator/icard/download-url` resolves in an immediate server rejection. They are not issued QR Codes. They do not exist on the public verification registry.
- **The Gamification Ban:** ENMs are deliberately excluded from the Gamified Offers infrastructure. They do not earn gamification points. They cannot see Active Offers. They do not occupy slots on the Monthly Leaderboard. Their output stats are purely absorbed into the visual representation of their supervising BDE. 
- **The Downline Ban:** ENMs sit at the bottom of the tree. They cannot recruit, spawn, or manage sub-users of any denomination.

# CHAPTER 8: THE PUBLIC-FACING ECOSYSTEM & ADMIN OVERSIGHT

The Andleeb Surya platform is not just an internal portal for agents and operators; it actively engages unauthenticated citizens across the region (Jammu & Kashmir, Ladakh) via a suite of robust Public Pages. The System Administrator acts as the ultimate curator of this public ecosystem.

## 8.1 The Public Route Map (No Auth Required)
The React Frontend exposes five primary unauthenticated routes designed for citizen interaction:
1. `GET /` — The main marketing Landing Page.
2. `GET /apply` — The Direct Citizen Application Portal.
3. `GET /track` — The Live Application Tracking Engine.
4. `GET /eligibility` — The Subsidy Calculation Tool.
5. `GET /verify/{token}` — The Public QR Verification Registry.

## 8.2 The Direct Application Portal (`/apply`)
Citizens do not need to wait for a field worker. They can navigate directly to the `/apply` route to submit their own `public_form` lead.
- **Data Capture:** They submit their `beneficiary_name`, `consumer_number` (globally unique to prevent spam), and heavily validated location details limited strictly to the 22 valid districts.
- **Referral Mechanics:** The form features an optional `referral_agent_id` field. If a citizen types in a BDE's code (e.g., SM-2026-1044), the lead instantly routes to that BDE's private pipeline pool.
- **Admin Failsafe (Orphan Prevention):** If a citizen leaves the referral field blank, OR if they type a fraudulent/suspended agent code, the lead is safely caught by the backend and deposited directly into the `admin_pool`. The Admin then utilizes the Bulk Assign engine to distribute these "organic" website leads as rewards to top-performing teams.

## 8.3 The Live Tracking Engine (`/track`)
To reduce customer support calls, the public ecosystem features a self-serve tracking portal.
- **Authentication Bypass:** A citizen does not log in. Instead, they provide a combination payload: their unique `consumer_number` paired with their `mobile` number. 
- **Rate-Limited Security:** To prevent malicious scraping of pipeline data, the `/api/v1/public/leads/track` endpoint is guarded by a strict firewall (`throttle:30,1` — maximum 30 attempts per minute per IP address). 
- **Visibility:** The citizen sees a simplified, consumer-friendly state machine showing exactly where their solar installation sits (e.g., Pending MNRE Subsidy, Installed, etc.).

## 8.4 The Subsidy Calculator (`/eligibility`)
A public financial tool utilizing the live J&K rate slab logic. A citizen selects their estimated capacity (e.g., 3kW) and instantly sees the breakdown: Total Project Cost (₹1,59,500) minus MNRE Subsidy (₹85,800) minus J&K State Subsidy (₹9,000) equalizing the exact "Citizen Pays" amount (₹64,700).

## 8.5 Admin Marketing & Testimonial Curation
The Admin wields absolute control over the social proof and branding rendered on these public pages.
- **Dynamic Variable Control:** Admins manipulate the `settings` database to instantly alter the frontend without redeploying code. They control the `login_overlay_title` ("Har Ghar Tak Surya"), the `login_overlay_subtitle` ("Malik Ke Saath"), and the public support lines.
- **The Beneficiary Survey Pipeline:** When a lead hits the `completed` status, a timed SMS/Email dispatches a signed URL valid for 7 days to the citizen. The citizen submits a 1-to-5 star rating and an optional roof photo/testimonial.
- **The Public Testimonial Toggle:** These surveys land privately on the Admin's dashboard. The Admin reviews them, discarding blurry photos or poor language, and presses "Publish" on 5-star reviews (`is_published=true`). These curated testimonials are then instantly beamed dynamically to the public Landing Page (`/`) to build massive brand trust.

## 8.6 The Security Registry (`/verify/{token}`)
To protect citizens from fraudulent actors claiming to represent the company, the public ecosystem hosts a cryptographically secure verification page.
- When an agent generates a PDF iCard, the backend generates a SHA-256 `qr_token`.
- A citizen scans the physical ID badge with their smartphone camera, routing them to the unauthenticated `/verify/{token}` page.
- **Visual Validation:** If valid, the page displays a massive Green Checkmark, the agent's photo, designation, and valid dates. 
- **The Admin "Kill Switch" Visualization:** If an Admin has pressed "Revoke" on an agent, this public URL instantly transforms into a massive Red "REVOKED" banner, alerting the citizen not to hand over cash or documents.
